# 《烟花大师》游戏设计总纲 (GDD)

---

## 一、项目概述 (Overview)

- **游戏类型**：烟花表演经营模拟 / 配置驱动策略 (Configuration-Driven)
- **核心体验**：收集配方 → 编排节目单 → 欣赏烟花表演 → 获取资金与名气 → 解锁更强配方 → 最终成为「烟花大师」
- **目标平台**：MVP 阶段为单页面 Web (HTML5/JS)，纯静态部署在 GitHub Pages
- **设计立意**：以轻松休闲的基调，让玩家体验从无名学徒到烟花大师的成长过程。烟花表演是核心视觉爽点，配方组合与事件匹配是核心策略点。
- **通关条件**：职业等级达到「大师」

---

## 二、核心玩法循环 (Core Loop)

```
[主界面 Hub]
    │
    ▼
[赛事列表] ── 选择 ──► [主线赛事] / [重复事件] / [保底事件]
    │
    ▼
[配方选择] ── 组建 ──► [节目单]（若干烟花弹）
    │
    ▼
[表演阶段] ── Canvas 粒子系统实时展示烟花
    │
    ▼
[评分结算] ── 加权匹配 + 组合奖励 + 重复惩罚
    │
    ▼
[获得资金 / 名气]
    │
    ▼
[配方实验室] ── 研发新配方 / 解锁新配方
    │
    ▼
[名气达标] ── 晋升下一职业等级 ── 回到 Hub
```

### 关键决策点
1. **参加什么事件**：活动赚钱、比赛赚名，预算和偏好不同。
2. **节目单怎么组**：在预算/数量限制内，选择最匹配事件偏好的配方。
3. **资金怎么花**：研发新配方 vs 参加活动成本 vs 留存备用。
4. **名气怎么升**：完成主线赛事，必要时刷重复事件。

## 三、状态与资源 (State & Resources)

### 玩家状态
| 字段 | 含义 | 初始值 |
|------|------|--------|
| `funds` | 资金 | 500 |
| `fame` | 名气 | 0 |
| `rank` | 当前职业等级 | `apprentice` |
| `unlockedRecipes` | 已解锁配方 ID 列表 | 初始 4 个 |
| `ownedRecipes` | 已研发配方 ID 列表 | 初始 4 个 |
| `completedMainEvents` | 已完成主线事件 ID 集合 | 空 |

### 双货币用途
- **资金 (`funds`)**
  - 支付参加比赛/活动的报名费（如有）
  - 支付研发配方的费用
  - 某些活动可超支，但收益可能无法覆盖成本
- **名气 (`fame`)**
  - 达到阈值后晋升职业等级
  - 解锁更高级的事件
  - 解锁更稀有的配方（配方解锁后仍需资金研发）

### 存档
- 使用 `localStorage` 自动存档，key 为 `firework_master_save_v1`
- Schema 包含：资金、名气、等级、已解锁配方、已拥有配方、已完成主线事件

## 六、评分系统 (Scoring System)

### 6.1 加权匹配
每个事件对 5 个核心属性有权重，权重总和为 100。

示例（县城花火大会）：
```js
preferences: {
  shape: { chrysanthemum: 25 },
  color: { blue: 10, white: 10 },
  height: { mid: 10, high: 5 },
  duration: { medium: 15, long: 10 },
  effect: { tail: 5, glitter: 5 }
}
```

每发烟花弹的匹配分：
```
match_score = Σ(属性命中该事件偏好权重)
```

### 6.2 基础分
节目单基础分 = 所有烟花弹匹配分之和 / 理论最大匹配分 × 100

理论最大匹配分 = 烟花弹数量 × 事件中单个属性的最大权重

### 6.3 组合奖励 (Combo Bonus)
鼓励节目单内出现特定组合：

| 组合名 | 触发条件 | 加分 |
|--------|----------|------|
| 年味 | 红色 ≥2 + 金色 ≥2 + crackle ≥1 | +8% |
| 浪漫 | 粉色 ≥2 + 心形 ≥2 + 低声响 | +8% |
| 压轴 | 最后 3 发为高空 + 长持续 | +10% |
| 多彩 | 包含 ≥5 种不同颜色 | +5% |
| 节奏 | 短/中/长持续各 ≥1 | +5% |

### 6.4 重复惩罚
同一配方使用次数超过阈值后，每次额外使用扣分：
- 使用 1-2 次：无惩罚
- 使用 3 次：该配方每发 -5%
- 使用 4 次及以上：该配方每发 -10%

### 6.5 最终得分
```
final_score = base_score × (1 + combo_bonus) × (1 - repeat_penalty)
```

### 6.6 收益计算
根据事件类型和最终得分：

```js
// 比赛
fame_gain = base_fame * (score / 100)
funds_gain = base_funds * (score / 100) * 0.5

// 活动
funds_gain = base_funds * (score / 100)
fame_gain = base_fame * (score / 100) * 0.5

// 保底事件
funds_gain = fixed_low_amount * (score / 100)
```

得分档位：
| 分数 | 评级 | 收益倍率 |
|------|------|----------|
| 90-100 | S | ×1.5 |
| 75-89 | A | ×1.2 |
| 60-74 | B | ×1.0 |
| 40-59 | C | ×0.7 |
| <40 | D | ×0.4 |

## 七、成长系统 (Progression)

### 7.1 职业等级
| 等级 | 名称 | 名气阈值 | 解锁内容 |
|------|------|----------|----------|
| 1 | 学徒 | 0 | 学徒级事件 + 初始配方 |
| 2 | 熟手 | 200 | 熟手级事件 + 熟手级配方 |
| 3 | 技师 | 800 | 技师级事件 + 技师级配方 |
| 4 | 名家 | 2500 | 名家级事件 + 名家级配方 |
| 5 | 大师 | 6000 | 大师级事件 + 大师级配方，通关 |

### 7.2 晋升规则
- 完成当前等级的所有主线赛事
- 名气达到下一等级阈值
- 同时满足以上两个条件后自动晋升

### 7.3 研发与解锁
- 配方因 `unlockFame` 达标而出现在「配方实验室」的解锁列表中
- 玩家支付 `researchCost` 后，该配方从「已解锁」变为「已拥有」
- 只有「已拥有」的配方才能在比赛中使用

## 八、烟花表演渲染 (Firework Rendering)

### 8.1 技术选型
- **Canvas 2D**：使用单个 `<canvas>` 元素绘制整个表演
- **粒子系统**：每个烟花弹爆炸后生成一组粒子
- **物理拟真**：
  - 重力影响粒子下落
  - 空气阻力使粒子减速
  - 随机初速度方向形成不同形状
  - 拖尾效果通过保留粒子历史位置实现

### 8.2 视觉映射
| 配方属性 | 渲染表现 |
|----------|----------|
| 颜色 | 粒子主色调 |
| 形状 | 粒子初始速度分布 pattern |
| 高度 | 烟花弹升空高度和爆炸高度 |
| 持续时间 | 粒子存活时间 |
| 特效 | 额外绘制效果 |

### 8.3 特效对应
| 特效 | 渲染表现 |
|------|----------|
| `none` | 标准爆炸 |
| `glitter` | 粒子闪烁，亮度周期性变化 |
| `crackle` | 粒子随机二次爆裂 |
| `strobe` | 粒子快速明暗变化 |
| `tail` | 升空阶段有明显拖尾 |
| `bouquet` | 爆炸后分裂成多个小花 |

### 8.4 表演流程
1. 根据当前事件的 `backdrop` 配置绘制表演背景（天空、星星、云层、地面剪影）
2. 按节目单顺序，每隔一定间隔发射一发烟花
3. 每发烟花：升空 → 最高点停顿 → 爆炸 → 粒子扩散下落 → 消失
4. 所有烟花播放完毕后进入评分结算

### 8.5 性能考虑
- 单屏粒子数控制在 300-500 以内
- 使用对象池复用粒子对象
- `requestAnimationFrame` 驱动动画
- 表演结束自动清理粒子

### 8.6 表演背景 (Backdrop)

为了增强表演临场感，每个事件可配置一套背景。表演开始时，`game.js` 把当前事件的 `backdrop` 传给 `FireworkRenderer`，渲染器据此绘制天空与地面。

#### 背景配置结构

```js
backdrop: {
  sky: 'midnight',     // 天空主题：midnight / twilight / festival / clear
  ground: 'village',   // 地面主题：village / grass / city / park
  clouds: 'few'        // 云层密度：none / few / scattered
}
```

#### 天空主题

| 主题 | 视觉效果 | 星星数量 | 适用场景 |
|---|---|---|---|
| `midnight` | 深夜蓝黑渐变 | 90 | 大型比赛、高级赛事 |
| `twilight` | 黄昏紫红渐变 | 55 | 浪漫活动、傍晚庆典 |
| `festival` | 节日暗红转深蓝 | 70 | 庙会、节日热闹场合 |
| `clear` | 晴朗深蓝渐变 | 110 | 日常表演、公园、保底事件 |

#### 地面主题

| 主题 | 视觉效果 | 适用场景 |
|---|---|---|
| `village` | 连续房屋屋顶剪影 | 庙会、夜市、乡村赛事 |
| `grass` | 缓坡草地丘陵 | 街头表演、开阔平地 |
| `city` | 高楼建筑剪影 | 市级/省级/国际城市赛事 |
| `park` | 草地 + 树木剪影 | 婚礼、公园、度假村 |

#### 云层

- `none`：无云
- `few`：1-2 朵缓慢飘动的薄云
- `scattered`：3-4 朵薄云

云层使用半透明白色圆簇绘制，每帧轻微水平移动，营造氛围但不抢烟花主体视觉。

#### 渲染策略

- 静态部分（天空渐变、星星、地面剪影）绘制到离屏 Canvas 缓存，尺寸变化时重建
- 星星使用基于种子的伪随机分布，避免与云层位置线性相关
- 动态云层每帧在主 Canvas 上绘制，叠加在静态背景之上、烟花之下
- 若当前事件未配置 `backdrop`，回退到原有深蓝/黑色渐变夜空

## 十一、技术架构

### 11.1 目录结构

```
firework-master/
├── index.html          # 游戏 DOM 容器
├── style.css           # 游戏样式
├── js/
│   ├── config.js       # 配方、事件、等级、组合规则等数据
│   ├── state.js        # 游戏状态 + 存档系统
│   ├── systems.js      # 评分系统、事件系统、成长系统
│   ├── renderer.js     # 烟花 Canvas 渲染器 + DOM 渲染
│   └── game.js         # 游戏主控 + 入口
└── doc/
    └── mvp-design.md   # 本设计文档
```

### 11.2 与 CyberTravel 的复用关系

| CyberTravel 模块 | Firework Master 复用情况 |
|------------------|--------------------------|
| `config.js` 数据驱动 | 复用模式，内容全部替换为配方/事件/等级数据 |
| `state.js` 状态 + 存档 | 复用结构，字段简化为资金/名气/配方/等级 |
| `systems.js` 计算系统 | 复用模块划分，新增评分系统、成长系统，移除天气/平台算法 |
| `renderer.js` DOM 渲染 | 复用模式，但界面结构大幅简化 |
| `game.js` 阶段机 | 复用状态机思想，阶段改为 Hub → 赛事 → 编队 → 表演 → 结算 |

### 11.3 技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| Canvas 粒子性能不足 | 低端设备卡顿 | 控制粒子数量，使用对象池，提供关闭特效选项 |
| 单文件 HTML 体积过大 | 首次加载慢 | MVP 内容控制在合理范围，图片/资源内联最小化 |
| 评分公式不平衡 | 某些配方/事件过强 | 先放占位数值，后续通过 playtest 调整 |
| 手机端交互体验差 | 触摸操作不便 | 主要按钮做大，编队界面支持左右分栏/堆叠适配 |
