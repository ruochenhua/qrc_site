# 《烟花大师》游戏设计总纲 (GDD)

---

## 一、项目概述 (Overview)

- **游戏类型**：烟花表演经营模拟 / 配置驱动策略 (Configuration-Driven)
- **核心体验**：解锁并购买材料 → 自由组装烟花 → 编排节目单 → 欣赏烟花表演 → 获取资金与名气 → 解锁更强材料 → 最终成为「烟花大师」
- **目标平台**：MVP 阶段为单页面 Web (HTML5/JS)，纯静态部署在 GitHub Pages
- **设计立意**：以轻松休闲的基调，让玩家体验从无名学徒到烟花大师的成长过程。烟花表演是核心视觉爽点，配方组合与事件匹配是核心策略点。
- **通关条件**：职业等级达到「大师」

---

## 二、核心玩法循环 (Core Loop)

```
[主界面 Hub]
    │
    ▼
[赛事列表] ── 选择 ──► [主线赛事] / [活动] / [重复事件] / [保底事件]
    │
    ▼
[组装台] ── 用已拥有材料组建烟花弹 ──► [节目单]
    │
    ▼
[表演阶段] ── Canvas 粒子系统实时展示烟花
    │
    ▼
[评分结算] ── 加权匹配 + 组合奖励 + 重复惩罚
    │
    ▼
[获得资金 / 名气]
    │
    ├──────► [活动] 偏重资金，可发射大量烟花
    │
    ├──────► [主线赛事] 多轮淘汰，按名次给名气，强制使用已拥有材料
    │
    ▼
[配方实验室] ── 用名气解锁可见性 + 用资金购买新材料
    │
    ▼
[名气达标] ── 晋升下一职业等级 ── 回到 Hub
```

### 关键决策点
1. **参加什么事件**：活动赚钱、比赛赚名，预算和偏好不同。
2. **节目单怎么组**：在预算/数量限制内，用已拥有的材料组装最匹配事件偏好的烟花弹。
3. **资金怎么花**：购买新材料 vs 参加活动成本 vs 留存备用。
4. **名气怎么升**：完成主线赛事，必要时刷重复事件或高名次比赛。

## 三、状态与资源 (State & Resources)

### 玩家状态
| 字段 | 含义 | 初始值 |
|------|------|--------|
| `funds` | 资金 | 800 |
| `fame` | 名气 | 0 |
| `rank` | 当前职业等级 | `apprentice` |
| `ownedComponents` | 已拥有材料 ID 集合 | 初始一套学徒材料 |
| `unlockedRecipes` | 已可见示例配方 ID 集合（保留字段，兼容旧档） | 初始 3 个 |
| `ownedRecipes` | 已拥有示例配方 ID 集合（保留字段，兼容旧档） | 初始 3 个 |
| `completedMainEvents` | 已完成主线事件 ID 集合 | 空 |

### 双货币用途
- **资金 (`funds`)**
  - 支付参加比赛/活动的报名费（如有）
  - 支付购买新材料的费用
  - 某些活动可超支，但收益可能无法覆盖成本
- **名气 (`fame`)**
  - 达到阈值后晋升职业等级
  - 解锁更高级的事件
  - 解锁更稀有的材料（材料可见后仍需资金购买）

### 存档
- 使用 `localStorage` 自动存档，key 为 `firework_master_save_v1`
- Schema 包含：资金、名气、等级、`ownedComponents`、已解锁示例配方、已拥有示例配方、已完成主线事件

## 四、组件与配方系统 (Components & Recipes)

### 4.1 核心定位
- **材料 (Component)** 是游戏中的最小可收集单位，分为火药、壳体、颜色剂、引线、特效剂五大类。
- **配方 (Recipe)** 退化为「系统示例蓝图」，展示一种可行的材料组合；玩家只要拥有配方所需的全部材料，就可以一键加载该组合到组装台。
- 玩家通过配方实验室购买材料，而非完整配方。

### 4.2 材料属性

#### 材料分类
| 分类 | 关键属性 | 说明 |
|------|----------|------|
| 火药 (gunpowder) | `thrust` | 决定升空高度与爆炸规模 |
| 壳体 (casing) | `shape`, `capacity`, `layers`, `scaleMultiplier` | 决定爆炸形状、容量与是否支持二层装药 |
| 颜色剂 (colorant) | `color`, `density` | 决定视觉颜色与粒子密度 |
| 引线 (fuse) | `length`, `heightFactor` | 影响升空高度与引爆时机 |
| 特效剂 (effect) | `effect`, `threshold`, `intensity` | 触发 glitter / crackle 等特效 |

#### 元属性（材料与示例配方共用）
| 属性 | 说明 |
|------|------|
| `cost` 成本 | 每单位/每发消耗的资金 |
| `unlockFame` 解锁名气 | 材料/配方出现在实验室列表所需的名气阈值 |
| `researchCost` 购买资金 | 购买材料所有权所需资金（配方示例无此花费） |
| `name` 名称 | 展示用名称 |
| `desc` 描述 | 一句话描述 |

#### 烟花弹派生属性
组装后每个烟花弹会派生以下属性：
| 属性 | 可选值 | 说明 |
|------|--------|------|
| `color` 颜色 | `red`, `gold`, `blue`, `green`, `white`, `purple`, `pink`, `silver`, `multi` | 主要视觉颜色 |
| `shape` 形状 | `peony`, `chrysanthemum`, `willow`, `ring`, `heart`, `star`, `palm`, `crackling` | 爆炸后呈现的形状 |
| `height` 高度 | `low`, `mid`, `high`, `ultra` | 升空高度 |
| `duration` 持续时间 | `short`, `medium`, `long` | 绽放后残影持续时长 |
| `effect` 特效 | `none`, `glitter`, `crackle`, `strobe`, `tail`, `bouquet` | 额外视觉特效 |

## 五、事件系统 (Event System)

### 5.1 事件类型

| 类型 | 收益偏向 | 预算 | 说明 |
|------|----------|------|------|
| **主线赛事** | 名气为主，资金为辅 | 硬预算，超支不能参加 | 每级必须完成才能晋升；采用多轮淘汰赛，按最终名次发放名气 |
| **活动** | 资金为主，名气为辅 | 软预算，可超支但可能亏本 | 可选参加；允许装载大量烟花弹（maxShells 远高于同等级赛事） |
| **重复事件** | 根据事件类型 | 根据事件类型 | 随机刷新，可重复参加 |
| **保底事件** | 稳定少量资金 | 无门槛/极低成本 | 防止卡死 |

### 5.2 事件属性
| 属性 | 说明 |
|------|------|
| `name` | 事件名称 |
| `type` | `competition` / `activity` / `repeatable` / `fallback` |
| `rank` | 所需职业等级 |
| `isMain` | 是否为主线赛事 |
| `budget` | 预算上限（比赛/部分活动） |
| `maxShells` | 最大烟花弹数量 |
| `entryFee` | 报名费 |
| `preferences` | 事件偏好权重（见评分系统） |
| `rewards` | 基础奖励（资金、名气） |
| `firstClearBonus` | 首次通关额外奖励 |
| `backdrop` | 表演背景配置（见 8.6 表演背景） |

### 5.3 赛事淘汰机制

- 所有 `competition` 类型事件采用**多轮淘汰赛**：
  - 学徒 / 熟手 / 技师：3 轮（8 人参赛）。
  - 名家 / 大师：4 轮（16 人参赛）。
- 每轮根据玩家节目单得分与系统生成的对手得分比较，胜者晋级。
- 最终名次决定名气奖励：
  - 第 1 名：100% 名气基础奖励
  - 第 2 名：60%
  - 第 3–4 名：35%
  - 第 5–8 名：15%
  - 第 9–16 名：5%
- 资金奖励随名次递减，且整体低于同等级活动。
- 结算页显示每轮胜负与最终名次。

### 5.4 活动大量发射

- 所有 `activity` 类型事件的 `maxShells` 显著高于同等级 `competition`，允许玩家发射大量烟花。
- 活动仍受玩家当前资金限制（单发成本累加），但预算为软约束，只影响最终收益不阻止参加。

## 六、评分系统 (Scoring System)

### 6.1 加权匹配
每个事件对 5 个核心属性有权重，权重总和为 100。

示例（县城花火大会）：
```js
preferences: {
  shape: { chrysanthemum: 25 },
  color: { blue: 10, white: 10 },
  height: { mid: 10, high: 5 },
  duration: { medium: 15, long: 10 },
  effect: { tail: 5, glitter: 5 }
}
```

每发烟花弹的匹配分：
```
match_score = Σ(属性命中该事件偏好权重)
```

### 6.2 基础分
节目单基础分 = 所有烟花弹匹配分之和 / 理论最大匹配分 × 100

理论最大匹配分 = 烟花弹数量 × 事件中单个属性的最大权重

### 6.3 组合奖励 (Combo Bonus)
鼓励节目单内出现特定组合：

| 组合名 | 触发条件 | 加分 |
|--------|----------|------|
| 年味 | 红色 ≥2 + 金色 ≥2 + crackle ≥1 | +8% |
| 浪漫 | 粉色 ≥2 + 心形 ≥2 + 低声响 | +8% |
| 压轴 | 最后 3 发为高空 + 长持续 | +10% |
| 多彩 | 包含 ≥5 种不同颜色 | +5% |
| 节奏 | 短/中/长持续各 ≥1 | +5% |

### 6.4 重复惩罚
同一配方使用次数超过阈值后，每次额外使用扣分：
- 使用 1-2 次：无惩罚
- 使用 3 次：该配方每发 -5%
- 使用 4 次及以上：该配方每发 -10%

### 6.5 最终得分
```
final_score = base_score × (1 + combo_bonus) × (1 - repeat_penalty)
```

### 6.6 收益计算
根据事件类型和最终得分：

```js
// 比赛
fame_gain = base_fame * (score / 100)
funds_gain = base_funds * (score / 100) * 0.5

// 活动
funds_gain = base_funds * (score / 100)
fame_gain = base_fame * (score / 100) * 0.5

// 保底事件
funds_gain = fixed_low_amount * (score / 100)
```

得分档位：
| 分数 | 评级 | 收益倍率 |
|------|------|----------|
| 90-100 | S | ×1.5 |
| 75-89 | A | ×1.2 |
| 60-74 | B | ×1.0 |
| 40-59 | C | ×0.7 |
| <40 | D | ×0.4 |

## 七、成长系统 (Progression)

### 7.1 职业等级
| 等级 | 名称 | 名气阈值 | 解锁内容 |
|------|------|----------|----------|
| 1 | 学徒 | 0 | 学徒级事件 + 初始材料包 + 3 个示例配方 |
| 2 | 熟手 | 200 | 熟手级事件 + 更多材料可见 |
| 3 | 技师 | 800 | 技师级事件 + 高级材料可见 |
| 4 | 名家 | 2500 | 名家级事件 + 稀有材料可见 |
| 5 | 大师 | 6000 | 大师级事件 + 顶级材料可见，通关 |

### 7.2 晋升规则
- 完成当前等级的所有主线赛事。
- 名气达到下一等级阈值。
- 同时满足以上两个条件后**自动晋升**，无需手动点击，并以 Toast 提示玩家。
- 达到「大师」等级后游戏通关。

### 7.3 材料解锁与购买
- 材料因 `unlockFame` 达标而出现在「配方实验室」的可见列表中。
- 玩家支付 `researchCost` 资金后，获得该材料的**所有权**。
- 只有「已拥有」的材料才能在组装台中使用；未拥有的可见材料会显示 🔒，不可点击选择。
- 示例配方同理：只要配方所需的全部材料均已拥有，玩家即可一键加载该示例到组装台。

## 八、烟花表演渲染 (Firework Rendering)

### 8.1 技术选型
- **Canvas 2D**：使用单个 `<canvas>` 元素绘制整个表演
- **粒子系统**：每个烟花弹爆炸后生成一组粒子
- **物理拟真**：
  - 重力影响粒子下落
  - 空气阻力使粒子减速
  - 随机初速度方向形成不同形状
  - 拖尾效果通过保留粒子历史位置实现

### 8.2 视觉映射
组装后的烟花弹派生属性到渲染表现：

| 派生属性 | 渲染表现 |
|----------|----------|
| 颜色 | 粒子主色调 |
| 形状 | 粒子初始速度分布 pattern |
| 高度 | 烟花弹升空高度和爆炸高度 |
| 持续时间 | 粒子存活时间 |
| 特效 | 额外绘制效果 |

### 8.3 特效对应
| 特效 | 渲染表现 |
|------|----------|
| `none` | 标准爆炸 |
| `glitter` | 粒子闪烁，亮度周期性变化 |
| `crackle` | 粒子随机二次爆裂 |
| `strobe` | 粒子快速明暗变化 |
| `tail` | 升空阶段有明显拖尾 |
| `bouquet` | 爆炸后分裂成多个小花 |

### 8.4 表演流程
1. 根据当前事件的 `backdrop` 配置绘制表演背景（天空、星星、云层、地面剪影）
2. 按节目单顺序，每隔一定间隔发射一发烟花
3. 每发烟花：升空 → 最高点停顿 → 爆炸 → 粒子扩散下落 → 消失
4. 所有烟花播放完毕后进入评分结算

### 8.5 性能考虑
- 单屏粒子数控制在 300-500 以内
- 使用对象池复用粒子对象
- `requestAnimationFrame` 驱动动画
- 表演结束自动清理粒子

### 8.6 表演背景 (Backdrop)

为了增强表演临场感，每个事件可配置一套背景。表演开始时，`game.js` 把当前事件的 `backdrop` 传给 `FireworkRenderer`，渲染器据此绘制天空与地面。

#### 背景配置结构

```js
backdrop: {
  sky: 'midnight',     // 天空主题：midnight / twilight / festival / clear
  ground: 'village',   // 地面主题：village / grass / city / park
  clouds: 'few'        // 云层密度：none / few / scattered
}
```

#### 天空主题

| 主题 | 视觉效果 | 星星数量 | 适用场景 |
|---|---|---|---|
| `midnight` | 深夜蓝黑渐变 | 90 | 大型比赛、高级赛事 |
| `twilight` | 黄昏紫红渐变 | 55 | 浪漫活动、傍晚庆典 |
| `festival` | 节日暗红转深蓝 | 70 | 庙会、节日热闹场合 |
| `clear` | 晴朗深蓝渐变 | 110 | 日常表演、公园、保底事件 |

#### 地面主题

| 主题 | 视觉效果 | 适用场景 |
|---|---|---|
| `village` | 连续房屋屋顶剪影 | 庙会、夜市、乡村赛事 |
| `grass` | 缓坡草地丘陵 | 街头表演、开阔平地 |
| `city` | 高楼建筑剪影 | 市级/省级/国际城市赛事 |
| `park` | 草地 + 树木剪影 | 婚礼、公园、度假村 |

#### 云层

- `none`：无云
- `few`：1-2 朵缓慢飘动的薄云
- `scattered`：3-4 朵薄云

云层使用半透明白色圆簇绘制，每帧轻微水平移动，营造氛围但不抢烟花主体视觉。

#### 渲染策略

- 静态部分（天空渐变、星星、地面剪影）绘制到离屏 Canvas 缓存，尺寸变化时重建
- 星星使用基于种子的伪随机分布，避免与云层位置线性相关
- 动态云层每帧在主 Canvas 上绘制，叠加在静态背景之上、烟花之下
- 若当前事件未配置 `backdrop`，回退到原有深蓝/黑色渐变夜空

## 十一、技术架构

### 11.1 目录结构

```
firework-master/
├── index.html          # 游戏 DOM 容器
├── css/                # 样式目录
│   ├── base.css        # 变量、重置、通用工具类
│   ├── views/          # 各视图专有样式
│   └── components/     # 跨视图组件样式
├── js/
│   ├── config.js       # 配方、事件、等级、组合规则等数据
│   ├── state.js        # 游戏状态 + 存档系统
│   ├── systems.js      # 评分系统、事件系统、成长系统
│   ├── renderer.js     # 烟花 Canvas 渲染器 + DOM 渲染
│   └── game.js         # 游戏主控 + 入口
└── doc/
    └── mvp-design.md   # 本设计文档
```

### 11.2 与 CyberTravel 的复用关系

| CyberTravel 模块 | Firework Master 复用情况 |
|------------------|--------------------------|
| `config.js` 数据驱动 | 复用模式，内容全部替换为配方/事件/等级数据 |
| `state.js` 状态 + 存档 | 复用结构，字段简化为资金/名气/配方/等级 |
| `systems.js` 计算系统 | 复用模块划分，新增评分系统、成长系统，移除天气/平台算法 |
| `renderer.js` DOM 渲染 | 复用模式，但界面结构大幅简化 |
| `game.js` 阶段机 | 复用状态机思想，阶段改为 Hub → 赛事 → 编队 → 表演 → 结算 |

### 11.3 技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| Canvas 粒子性能不足 | 低端设备卡顿 | 控制粒子数量，使用对象池，提供关闭特效选项 |
| 单文件 HTML 体积过大 | 首次加载慢 | MVP 内容控制在合理范围，图片/资源内联最小化 |
| 评分公式不平衡 | 某些配方/事件过强 | 先放占位数值，后续通过 playtest 调整 |
| 手机端交互体验差 | 触摸操作不便 | 主要按钮做大，编队界面支持左右分栏/堆叠适配 |
