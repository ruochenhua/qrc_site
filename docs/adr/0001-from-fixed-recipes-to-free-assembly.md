# ADR 0001: 从固定配方转向自由组件组装

## 状态

已决定

## 背景

Firework Master 最初的 MVP 设计采用「固定配方」机制：玩家通过名气解锁 20 个预设配方，消耗资金研发后，在比赛中挑选若干配方组成表演。评分基于配方属性与事件偏好的离散匹配。

随着原型推进，发现这一机制存在以下问题：

- 玩家实际可做的选择很少，核心玩法变成「选几个配方」，而非「设计烟花」
- 表演视觉效果缺乏差异化，不同配方看起来区别不大
- 配方作为固定正确答案，限制了玩家创造力和重玩价值

因此决定将核心玩法从「解锁配方」转向「解锁材料并自由组装」。

## 决策

采用 **Kerbal Space Program 式的自由组装模型**：

1. 玩家解锁的是**组件/材料**，不是固定配方
2. 组件按**数量**添加，数量影响最终烟花的物理表现
3. 壳体提供**容量上限**和**形状基础**
4. 多层壳支持**二次爆炸**
5. 系统配方退化为**示例蓝图**，仅作引导
6. 玩家可保存自己的设计为**蓝图**
7. 评分从离散标签匹配改为**连续属性拟合**

## 关键规则

| 组件 | 作用 |
|------|------|
| 火药 | 数量决定升空高度、爆炸规模、粒子数 |
| 壳体 | 决定形状、容量上限、成本；多层壳支持二次爆炸 |
| 颜色剂 | 数量决定粒子密度；多种颜色剂按比例分配粒子颜色 |
| 引线 | 长度决定爆炸高度（短=低空，长=高空） |
| 特效剂 | 达到阈值触发对应特效；二次爆炸药+多层壳触发多层绽放 |

## 权衡

### 放弃的替代方案

- **保留固定配方，只扩展数量**：改动最小，但无法解决核心玩法单薄的问题
- **完全涌现式形状**：由装药方式决定爆炸形状，更 KSP，但对 MVP 过于硬核

### 选择的理由

- 自由组装是游戏的灵魂，MVP 必须把这部分做对
- 组件数量制既真实又有涌现空间，同时规则简单易懂
- 壳体决定形状 + 多层壳支持二次爆炸，在真实性和可玩性之间取得平衡

## 后果

- 需要重写 `config.js` 中的组件数据和 `systems.js` 中的组装、评分逻辑
- 需要新增蓝图保存/加载系统
- 现有固定配方加成机制需要移除或替换
- 渲染器需要根据新的连续属性（高度、规模、粒子密度、二次爆炸）进行调整
- 数值平衡需要重新设计
